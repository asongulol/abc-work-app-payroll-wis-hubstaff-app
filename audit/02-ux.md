## Consolidation candidates

No screens warrant a merge — the IA is already disciplined (5 sidebar groups, each ≤6 items, within the ≤7 guideline `:20`). Two relationships are worth a deliberate decision rather than a merge:

1. **`process` (Process and Pay) ↔ `batches` (Review & Recon Batches).** Same component (`ProcessPayroll`) mounted twice — once normal, once `reconcileOnly` (`13634`,`13636`). The code comment (`9989`) states this is intentional: reconcile is split out into the Review group to keep the per-run Process screen uncluttered. **Recommendation: keep the split, do NOT consolidate**, but make the two sidebar labels unmistakably different tasks (today "Process and Pay" vs "Review & Recon Batches" — adequate). Flagged only so a future reviewer doesn't mistake it for accidental duplication.

2. **AddContractorWizard ↔ ProfileModal "Pay & payout".** Both edit the same hire/rate/payout fields (the wizard creates; the modal edits — OnboardingDetail even reuses ProfileModal for "Edit details", `11438`). Already deduplicated at the component level; no merge needed. Proposed shape if either is ever reworked: a single full-page contractor form (create vs edit mode) so the create path stops being modal-bound (see Removal/Revise note below).

## Accessibility & interaction-state findings (cross-cutting)

1. **Portal has NO visible focus styles and NO `:hover`/`:active`/`:focus-visible` rules** (grep: 0 occurrences of `:focus`/`outline` in `portal/index.html`; admin has 9 incl. the global `:focus-visible{outline:2px solid var(--navy)}` at `app/index.html:72`). The portal relies entirely on browser-default focus rings, which several custom controls suppress. **Add a portal `:focus-visible` ring** mirroring the admin token. Guideline `:148` (visible focus indicators) — required, not optional.

2. **Portal modals lack focus management.** Admin shipped `useModalA11y` (focus-trap + restore + Escape + modal stack, wired to ~25 modals — `app/index.html:11407`, confirmed in `ux-ui-improvements.md`). The portal has **no equivalent**: AgreementViewer (`1815`), DocReminder (`1242`), mood check-in (`1564`), ToolsPopup (`2210`) declare `role="dialog"`/`aria-modal` but have no focus-trap, no focus-restore, and (except UnsavedGuardModal `492`) no Escape handler. Keyboard/screen-reader users can tab into the page behind an open dialog. **Port `useModalA11y` to the portal** (UnsavedGuardModal `485-499` is the in-repo template).

3. **`<div onClick>` / `<span role=button>` controls that aren't keyboard-operable** (portal): PaySlips expander (`765`), TimeView expander (`833`), mood emojis (`1570`). They have `cursor:pointer`/`role="button"` but no `tabIndex` and no `onKeyDown` for Enter/Space. Guideline `:54` (interactive looks/acts interactive) + `:148` (keyboard nav). Fix: make them real `<button>`s or add `tabIndex={0}` + key handler.

4. **Native `window.confirm` for destructive actions** persists in **27 admin** call-sites and **2 portal** (`grep`), e.g. remove-admin (`app/index.html:1368`), delete-announcement (`2164`). The owner shipped a styled typed-confirm host (`DangerConfirm`/`confirmDanger`) for the high-stakes money/batch actions, but the long tail still uses the OS dialog — which breaks the navy/gold visual system (Consistency `:144`) and offers no themed danger affordance. **Migrate remaining `window.confirm` to the styled `DangerConfirm` pattern.** (Money-critical paths already use typed-confirm — this is a consistency/polish gap, not a money-safety hole.)

5. **Loading: skeletons vs text spinners (portal).** Admin uses loading skeletons that reserve layout (`app/index.html:709`, 28 uses) per guideline `:33`/`:134` (skeleton before data, avoid layout shift). The **portal uses plain `"Loading…"` text** in every view (`portal/index.html:744,808`, 0 skeleton uses) — content jumps in when data arrives. For PaySlips/TimeView/Sessions/Docs this is a layout-shift + perceived-speed regression vs the standard. **Add card skeletons to the portal lists.**

6. **Color & contrast — strong, with one residual risk.** The token set was given a deliberate AA pass — `--subtle` carries a code comment "darkened … WCAG AA 4.5:1 (was 3.06)" (`app/index.html:48`) and `--muted:#5c6677` passes on white. This is exemplary. Residual watch-items: **gold `#D4A24C` as text on white fails AA** (~2.0:1) — verify it is only used for large text/accents/borders, never small body text (guideline `:108-109`, color-not-alone `:151`). Status pills (paid/pending in portal `769`) should pair color with text/icon — they currently do carry text labels, which satisfies `:109`.

7. **`prefers-reduced-motion` supported** in both apps (admin 6 blocks, portal 2) — satisfies `:129`. Good.

8. **Empty-state quality (spot-check needed).** Both apps have empty markers (admin 20, portal 22), and portal empties are friendly ("No pay slips yet." `745`). Guideline `:64` wants empties to *offer the first action* — admin maintenance screens (Documents `7467`, Imports `4971`) should be checked that they don't dead-end with a bare "no rows" and instead point to the action (e.g. "Import time to begin"). OBSERVED present; quality of copy = INFERRED, recommend a sweep.
