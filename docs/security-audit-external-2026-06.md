# Security Audit of `abc‑work‑app‑payroll‑wis‑hubstaff‑app`

This security review examined all accessible code in the repository (serverless functions, Supabase policies, client code, tooling and documentation) using the checklist of injection issues, authentication/authorization (AuthN/AuthZ), secret handling and dependencies.  Supabase Edge functions are written for Deno and run on Supabase’s infrastructure.  The front‑end is a compiled React SPA served from `app/index.html` and an older `app/legacy.html`.  The audit treats the latest code (phase‑2 RLS) as the primary implementation but flags legacy files for deprecation.

## Authentication/Authorization Assessment

* **Admin‑only gating:**  Most serverless functions check for a bearer token and verify the caller is present in the `admin_users` table; non‑admin callers receive 403s [oai_citation:11‡raw.githubusercontent.com](https://raw.githubusercontent.com/asongulol/abc-work-app-payroll-wis-hubstaff-app/main/supabase/functions/hubstaff-sync/index.ts#:~:text=auth,).  Owner‑only actions such as funding Wise transfers require `role = 'owner'` [oai_citation:12‡raw.githubusercontent.com](https://raw.githubusercontent.com/asongulol/abc-work-app-payroll-wis-hubstaff-app/main/supabase/functions/wise-payouts/index.ts).  Machine actions accept a cron secret stored in `app_secrets`.  These checks reduce risk of unauthorized operations but rely on correct population of the `admin_users` table and secure delivery of cron secrets.
* **RLS policies:**  Phase‑2 migrations define row‑level security functions like `my_worker_id()` and `is_admin()` and restrict reads/writes accordingly [oai_citation:13‡raw.githubusercontent.com](https://raw.githubusercontent.com/asongulol/abc-work-app-payroll-wis-hubstaff-app/main/schema/migrations/2026-05-30_contractor_portal_rls.sql) [oai_citation:14‡raw.githubusercontent.com](https://raw.githubusercontent.com/asongulol/abc-work-app-payroll-wis-hubstaff-app/main/schema/migrations/2026-05-30_phase2_rls_admin.sql).  Only admins can read or modify critical tables; contractors can only read their own data.  This design mitigates the earlier vulnerability where shipping the anon key exposed the entire DB.
* **Legacy endpoints:**  The `legacy.html` file uses the old client with permissive RLS and should be removed to prevent bypassing the new policies.  Supabase’s row‑level policies apply at the database layer, so as long as `legacy.html` is removed, the anon key cannot perform privileged operations.

## Secrets and Key Management

* **Supabase anon key:**  The compiled SPA stores the public `SUPABASE_ANON` key in local storage and uses it for client queries.  This is acceptable because RLS policies restrict what an anon user can see [oai_citation:15‡raw.githubusercontent.com](https://raw.githubusercontent.com/asongulol/abc-work-app-payroll-wis-hubstaff-app/refs/heads/main/app/index.html#:~:text=placeholder%3D%7Bplaceholder%7C%7C).  **No service‑role key is exposed** in client code—serverless functions read it from environment variables [oai_citation:16‡raw.githubusercontent.com](https://raw.githubusercontent.com/asongulol/abc-work-app-payroll-wis-hubstaff-app/main/supabase/functions/hubstaff-sync/index.ts#:~:text=after%20each%20exchange,Type).
* **Service‑role key usage:**  Edge functions fetch data with `SUPABASE_SERVICE_ROLE_KEY`.  This key provides full DB access, so functions correctly gate calls behind admin checks [oai_citation:17‡raw.githubusercontent.com](https://raw.githubusercontent.com/asongulol/abc-work-app-payroll-wis-hubstaff-app/main/supabase/functions/hubstaff-sync/index.ts#:~:text=auth,) [oai_citation:18‡raw.githubusercontent.com](https://raw.githubusercontent.com/asongulol/abc-work-app-payroll-wis-hubstaff-app/main/supabase/functions/wise-payouts/index.ts).  Ensure that service keys are stored as Supabase secrets and not committed.
* **Cron secret:**  The cron secret is stored in `app_secrets` and presented via header.  It should be rotated regularly and transmitted only over HTTPS.
* **Wise API token and private key:**  The `WISE_API_TOKEN` and `WISE_PRIVATE_KEY` are not present in the repo and are loaded from environment variables [oai_citation:19‡raw.githubusercontent.com](https://raw.githubusercontent.com/asongulol/abc-work-app-payroll-wis-hubstaff-app/main/supabase/functions/wise-payouts/index.ts).  This is good practice.  Validate that the private key is stored securely and that SCA is enforced.

## Dependency Review

* **React & react‑dom (18.2.0):** No direct CVEs in these versions [oai_citation:20‡security.snyk.io](https://security.snyk.io/package/npm/react/18.2.0#:~:text=Direct%20Vulnerabilities), but a major supply‑chain compromise in June 2026 affected the npm registry (“Mastra”).  The repository loads React from `cdn.jsdelivr.net`, which was not compromised.  Still, monitor for new advisories.
* **Babel‑standalone (7.23.2):** This version includes the fix for CVE‑2023‑45133 (arbitrary code execution during compilation) [oai_citation:21‡github.com](https://github.com/advisories/GHSA-67hx-6x53-jw92#:~:text=crafted%20malicious%20code).  Avoid compiling untrusted code in the browser.
* **Denomailer (1.6.0):** The SMTP client has little community scrutiny and may be missing security updates.  Consider switching to `std/smtp` or a managed email provider.
* **Other scripts:** Shell and mjs scripts under `tools/` perform deployment, secret checks and mobile smoke tests.  They do not expose secrets directly but should be restricted to trusted users.

## Overall Recommendations

1. **Remove or restrict legacy client:** Delete `app/legacy.html` or protect it behind admin authentication.  Confirm that no build process regenerates it.
2. **Escape all user data in emails:** Refactor `documents‑expiry‑check`, `portal‑admin` and other functions to escape titles, names and comments before inserting into HTML emails.  Use a shared `escapeHtml()` helper.
3. **Validate and encode query parameters:** When building PostgREST queries, ensure user‑controlled values (IDs, names, dates) are validated (UUID or numeric) and encoded via `URLSearchParams`.  Do not construct `...&id=eq.${body.id}` with untrusted input.
4. **Update dependencies regularly:** Monitor React, Babel and Deno packages for advisories.  Replace `denomailer` with a maintained SMTP library or a third‑party email service.
5. **Rotate and protect secrets:** Use HTTPS for all cron and admin calls.  Rotate cron secrets periodically and store them securely.  Audit environment variables to ensure no secrets are accidentally logged.
6. **Implement structured logging:** Avoid logging raw error messages; sanitize or structure logs to prevent log injection.  Remove any sensitive data from logs.

These changes will improve the security posture of the HR & Payroll application by reducing injection points, strengthening authentication controls and protecting sensitive information.
